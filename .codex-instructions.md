# Codex Instructions for video-audit-electron

## Context & Problem

This repo is a new private macOS Electron version of an existing app named `video-audit`.

The existing legacy app lives in a sibling VS Code workspace folder named `video-audit`. It is a Vite + React + PrimeReact frontend with a local Node/Express backend. It audits local videos with ffprobe/ffmpeg, identifies low-resolution or wrong-aspect-ratio videos, detects black borders, can generate thumbnails, can auto-fix/crop videos, and can hand selected videos to a Premiere Pro bridge.

The new repo is `video-audit-electron`.

## Critical Repo Boundaries

- You may inspect the sibling `video-audit` repo for reference.
- You must only modify files inside `video-audit-electron`.
- Do not modify the legacy `video-audit` repo.
- Do not import from the legacy repo.
- Do not create cross-repo dependencies.
- The new Electron app must be standalone.

## Architecture Direction

The new app should use:

- Electron
- Vite
- React
- TypeScript
- PrimeReact
- Electron main process for filesystem access, ffprobe/ffmpeg, job orchestration, app settings, and OS integration
- Electron preload with a small typed API exposed via `contextBridge`
- Renderer for UI only
- Shared types in `src/shared`

The target architecture should not rely on Express or HTTP unless a stage explicitly says otherwise.

## Code Style

- Prefer simple, maintainable TypeScript.
- Avoid premature abstraction.
- Use explicit names.
- Keep Electron IPC channels typed and centralized.
- Do not expose raw Node APIs to the renderer.
- Keep `contextIsolation` enabled.
- Renderer must not directly use `fs`, `path`, `child_process`, or other Node APIs.

## Do NOT write tests

Do not add tests, test frameworks, test files, or test setup unless explicitly asked.

## Contribution Workflow

Follow `CONTRIBUTING.md` for changelog, versioning, commit, safety, dependency, and documentation rules.

When you make code, configuration, or documentation changes:

- Update `CHANGELOG.md`.
- Bump `package.json` version to match the latest changelog version when `package.json` exists.
- Stage all changes.
- Commit with a clear contextual commit message.

Do not push, tag, amend, rebase, or force-push unless explicitly asked.
